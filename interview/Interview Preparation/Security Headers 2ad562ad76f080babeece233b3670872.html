Security Headers

Understand the Security Patterns and Headers information in depth

Security headers are HTTP response headers that provide additional security measures to web applications. They are used to enhance the security posture of a website by enabling certain security features or mitigating common web vulnerabilities. Here are the types of security headers along with detailed examples:

Active Security Headers

Content-Security-Policy (CSP)

The Content-Security-Policy header defines the content sources that the browser should consider valid for a specific web page. It is one of the most powerful security headers for preventing XSS attacks and other injection vulnerabilities.

Example:

Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; img-src 'self' data:

Explanation

  • default-src 'self': Specifies that content can only be loaded from the same origin as the website itself.
  • script-src 'self' 'unsafe-inline': Allows JavaScript to be loaded from the same origin and permits inline scripts (generally not recommended for security reasons).
  • img-src 'self' data:: Allows images from the same origin and data URIs.

Other Examples

  • Using Nonces

    Nonces (number used once) are cryptographically secure random values that can be used to allow specific inline scripts:

    Content-Security-Policy: script-src 'self' 'nonce-abc123';
    <script nonce="abc123">
  // Safe, allowed script
</script>

  • Using Hashes

    Hashes can be used to allow specific inline scripts by including the hash of the script content:

    Generate Hash

    echo -n "console.log('This is a trusted inline script.');" | openssl dgst -sha256 -binary | openssl base64
    Content-Security-Policy: script-src 'self' 'sha256-IlQrFJU+ggN6Eys8sLDHPP8sujHssEqCzhWsu9YTHHo=';

  • Frame Ancestors Directive

    The frame-ancestors directive in CSP replaces the deprecated X-Frame-Options header:

    Content-Security-Policy: frame-ancestors 'none';  // Equivalent to X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'self';   // Equivalent to X-Frame-Options: SAMEORIGIN
Content-Security-Policy: frame-ancestors https://example.com;  // Allow specific origin

Strict-Transport-Security (HSTS)

The HTTP Strict Transport Security header enforces the use of HTTPS by instructing the browser to only communicate with the website over a secure connection.

Example:

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

Explanation:

  • max-age=31536000: Instructs the browser to enforce HTTPS for 31,536,000 seconds (one year).
  • includeSubDomains: Applies the HSTS policy to all subdomains.
  • preload: Indicates that the site should be included in browser preload lists for HSTS (requires submission to HSTS preload list).

X-Content-Type-Options

The X-Content-Type-Options header prevents content type sniffing (MIME sniffing), which can help mitigate certain types of attacks.

Example:

X-Content-Type-Options: nosniff

Explanation:

  • This value instructs the browser to strictly follow the declared content type and not perform content sniffing or guessing. It prevents the browser from trying to interpret the content based on its own heuristics, which can be exploited by attackers to trick the browser into executing malicious code.

Common Content Type

  • text/plain: Plain text content
  • text/html: HTML content for rendering web pages
  • text/css: Cascading Style Sheets (CSS) content
  • application/json: JSON (JavaScript Object Notation) data format
  • application/xml: XML (eXtensible Markup Language) content
  • application/pdf: Portable Document Format (PDF) file
  • image/jpeg: JPEG image format
  • image/png: PNG (Portable Network Graphics) image format
  • image/gif: GIF (Graphics Interchange Format) image format
  • audio/mpeg: MP3 audio format
  • video/mp4: MP4 video format
  • application/octet-stream: Binary data or unknown file format
  • multipart/form-data: Used in form submissions with file uploads
  • application/x-www-form-urlencoded: Default content type for form submissions
  • application/javascript: JavaScript code

Referrer-Policy

The Referrer-Policy header controls how much referrer information should be sent with requests from a web page.

Example:

Referrer-Policy: strict-origin-when-cross-origin

Available Directives

  • no-referrer: Completely omits the referrer information; no referrer information is sent with the request.
  • no-referrer-when-downgrade: Default behavior if the header is not specified. Sends referrer information when navigating from HTTPS to HTTPS, but omits it when navigating from HTTPS to HTTP.
  • same-origin: Referrer information is sent only for requests originating from the same origin (same domain and protocol). Not sent for cross-origin requests.
  • origin: Sends only the origin of the referrer URL (protocol, host, and port), excluding the path and query parameters.
  • strict-origin: Similar to origin, but only sends the origin if the protocol of the target resource matches the protocol of the referring page.
  • origin-when-cross-origin: Sends full referrer information when making requests within the same origin. For cross-origin requests, only sends the origin of the referrer URL.
  • strict-origin-when-cross-origin: Sends full referrer information for same-origin requests. For cross-origin requests, only sends the origin if the target resource is also secure (HTTPS).
  • unsafe-url: Sends the complete referrer information, including the path, query parameters, and fragment identifier, regardless of the protocol or origin.

Permission-Policy (formerly Feature-Policy)

The Permissions-Policy header allows websites to control which browser features can be used and by whom, enhancing privacy and security.

Example:

Permissions-Policy: geolocation=(), microphone=(), camera=(), payment=()

Explanation:

  • geolocation=(): Disables geolocation access
  • microphone=(): Disables microphone access
  • camera=(): Disables camera access
  • payment=(): Disables Payment Request API

Other Features:

  • accelerometer, gyroscope, magnetometer: Motion sensors
  • usb: USB device access
  • bluetooth: Bluetooth device access
  • fullscreen: Fullscreen API
  • picture-in-picture: Picture-in-Picture API

Cross-Origin-Opener-Policy (COOP)

COOP ensures that a top-level document does not share a browsing context group with cross-origin documents, preventing potential cross-origin attacks.

Example:

Cross-Origin-Opener-Policy: same-origin

Available Values:

  • same-origin: The document can only share a browsing context group with documents from the same origin.
  • same-origin-allow-popups: Allows popups to share the browsing context group if they are from the same origin.
  • unsafe-none: Default value; allows cross-origin documents to share the browsing context group.

Cross-Origin-Embedder-Policy (COEP)

COEP prevents a document from loading any cross-origin resources that do not explicitly grant permission, enhancing security against cross-origin attacks.

Example:

Cross-Origin-Embedder-Policy: require-corp

Available Values:

  • require-corp: Requires all resources to be loaded with Cross-Origin-Resource-Policy (CORP) headers or be same-origin.
  • credentialless: Allows loading cross-origin resources without credentials.
  • unsafe-none: Default value; allows loading cross-origin resources without restrictions.

Cross-Origin-Resource-Policy (CORP)

CORP prevents other domains from loading resources from your site, mitigating certain cross-origin attacks.

Example:

Cross-Origin-Resource-Policy: same-origin

Available Values:

  • same-origin: Only same-origin requests can load the resource.
  • same-site: Same-site requests (same domain or subdomain) can load the resource.
  • cross-origin: Allows cross-origin requests to load the resource.

Clear-Site-Data

The Clear-Site-Data header instructs the browser to clear browsing data (e.g., cookies, storage) associated with the requesting website.

Example:

Clear-Site-Data: "cache", "cookies", "storage"

Available Directives:

  • "cache": Clears cached data
  • "cookies": Clears cookies
  • "storage": Clears storage (localStorage, sessionStorage, etc.)
  • "executionContexts": Clears execution contexts (service workers, etc.)

Deprecated Headers

X-XSS-Protection (DEPRECATED)

Status: This header has been deprecated by modern browsers and is no longer recommended.

Previous Usage:

X-XSS-Protection: 1; mode=block

Reason for Deprecation: 

Modern browsers have deprecated this header in favor of more comprehensive measures like Content-Security-Policy (CSP), which provides better protection against XSS attacks. CSP offers more granular control and is actively maintained.

Recommendation: Use Content-Security-Policy instead.

X-Frame-Options (DEPRECATED in favor of CSP)

Status: While still supported, this header is being deprecated in favor of the frame-ancestors directive in CSP.

Previous Usage:

X-Frame-Options: DENY
X-Frame-Options: SAMEORIGIN
X-Frame-Options: ALLOW-FROM https://example.com

Note: The ALLOW-FROM directive was never widely supported and is not recommended.

Recommendation: Use the frame-ancestors directive in Content-Security-Policy instead:

Content-Security-Policy: frame-ancestors 'none';  // Instead of X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'self';  // Instead of X-Frame-Options: SAMEORIGIN

Expect-CT (DEPRECATED)

Status: This header has been deprecated.

Previous Usage:

Expect-CT: max-age=86400, enforce, report-uri="https://example.com/report"

Reason for Deprecation: Certificate Transparency is now enforced by browsers through other mechanisms, making this header unnecessary.

Possible Vulnerabilities and Mitigations

Cross-Site Scripting (XSS)

Description: XSS attacks occur when an attacker injects malicious scripts into web pages viewed by users.

Mitigation:

  • Content-Security-Policy (CSP): Specify allowed sources for content, preventing execution of unauthorized scripts.
  • X-Content-Type-Options: Prevent MIME sniffing that could lead to script execution.
  • Input Validation and Output Encoding: Always validate and sanitize user input.

Cross-Site Request Forgery (CSRF)

Description: CSRF attacks trick authenticated users into unknowingly performing actions on a website without their consent.

Mitigation:

  • CSRF Tokens: Generate a unique token for each user session and include it in each request that modifies state or performs sensitive actions. Validate the token on the server side.
  • SameSite Cookie Attribute: Set cookies with SameSite=Strict or SameSite=Lax to prevent cross-site requests from including cookies.
  • Custom Headers: Use custom headers (like X-Requested-With) that cannot be set by browsers in cross-origin requests.

Clickjacking

Description: Clickjacking involves tricking users into clicking on malicious elements disguised as legitimate buttons or links.

Mitigation:

  • Content-Security-Policy frame-ancestors: Use frame-ancestors 'none' or frame-ancestors 'self' to prevent framing.
  • X-Frame-Options: While deprecated, can still be used as a fallback: X-Frame-Options: DENY or X-Frame-Options: SAMEORIGIN.

MIME Sniffing Attacks

Description: Browsers may attempt to guess the content type of a resource, which can lead to security vulnerabilities.

Mitigation:

  • X-Content-Type-Options: nosniff: Prevents browsers from MIME-sniffing responses.

Protocol Downgrade Attacks

Description: Attackers may attempt to force a connection to use HTTP instead of HTTPS.

Mitigation:

  • Strict-Transport-Security (HSTS): Enforces HTTPS connections and prevents protocol downgrade attacks.

Information Disclosure

Description: Sensitive information may be leaked through referrer headers or other mechanisms.

Mitigation:

  • Referrer-Policy: Control how much referrer information is sent with requests.
  • Permissions-Policy: Restrict access to sensitive browser features like geolocation, camera, and microphone.

Best Practices Summary

  1. Always use HTTPS and implement HSTS with preload for maximum security.
  1. Implement CSP with a restrictive policy, using nonces or hashes for inline scripts.
  1. Use X-Content-Type-Options: nosniff to prevent MIME sniffing.
  1. Set appropriate Referrer-Policy based on your privacy requirements.
  1. Use Permissions-Policy to restrict unnecessary browser features.
  1. Consider COOP and COEP for enhanced isolation in modern applications.
  1. Avoid deprecated headers like X-XSS-Protection and prefer CSP alternatives.
  1. Use CSP's frame-ancestors instead of X-Frame-Options for clickjacking protection.
  1. Test your headers using tools like SecurityHeaders.com or Mozilla Observatory.
  1. Monitor and update your security headers regularly as browser standards evolve.

Testing Your Security Headers

You can test your security headers using:

  • SecurityHeaders.com: Provides a security header analysis
  • Mozilla Observatory: Comprehensive security analysis tool
  • Browser DevTools: Check Network tab to view response headers
