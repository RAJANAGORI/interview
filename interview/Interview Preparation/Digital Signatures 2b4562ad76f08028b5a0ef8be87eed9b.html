Introduction

What is a Digital Signature?

A digital signature is a cryptographic technique that provides:
- Authentication: Confirms who sent the message
- Integrity: Confirms the message hasn’t been altered
- Non-repudiation: Prevents the sender from denying they sent it

Why Digital Signatures Matter

Problems They Solve:
1. Impersonation: How do you know who really sent the message?
2. Tampering: How do you know the message wasn’t modified?
3. Repudiation: How do you prove someone sent something?

Real-World Analogy:
- Like a handwritten signature on paper
- But cryptographically secure
- Cannot be forged (computationally infeasible)
- Works for digital documents

Key Properties

  1. Authenticity: Verifies the identity of the signer
  1. Integrity: Detects any modification to the document
  1. Non-repudiation: Sender cannot deny sending
  1. Unforgeability: Cannot be created without private key

What is a Digital Signature?

Definition

A digital signature is a mathematical scheme for verifying the authenticity and integrity of digital messages or documents using public key cryptography.

Components

  1. Message/Document: The content being signed
  1. Hash Function: Creates a digest of the message
  1. Private Key: Used to sign the hash
  1. Public Key: Used to verify the signature
  1. Digital Signature: The signed hash value

How It Differs from Electronic Signatures

Electronic Signature:
- Broad term for any electronic method of signing
- Can be as simple as typing your name
- May not provide cryptographic security
- Legal concept

Digital Signature:
- Specific cryptographic technique
- Uses public key cryptography
- Provides mathematical proof of authenticity
- Technical implementation

Relationship:
- Digital signatures are a type of electronic signature
- Not all electronic signatures are digital signatures
- Digital signatures provide stronger security guarantees

How Digital Signatures Work

Creation Process (Sender’s Side)

Step 1: Hash the Document

Document → Hash Function → Hash Value (Digest)

Step 2: Sign the Hash

Hash Value → Private Key → Digital Signature

Step 3: Attach Signature

Document + Digital Signature → Send to Recipient

Visual Flow:

┌─────────────┐
│  Document   │
│  "Hello"    │
└──────┬──────┘
       │
       │ Hash Function (SHA-256)
       ▼
┌─────────────┐
│  Hash:      │
│  2cf24d...  │
└──────┬──────┘
       │
       │ Private Key (Encrypt)
       ▼
┌─────────────┐
│  Signature: │
│  ABC123...  │
└─────────────┘

Verification Process (Receiver’s Side)

Step 1: Receive Document and Signature

Document + Digital Signature

Step 2: Hash the Received Document

Document → Hash Function → Hash Value (Computed)

Step 3: Decrypt the Signature

Digital Signature → Public Key → Hash Value (From Signature)

Step 4: Compare Hashes

If Computed Hash == Hash From Signature:
    ✅ Signature is VALID
    ✅ Document is authentic and unmodified
Else:
    ❌ Signature is INVALID
    ❌ Document may be tampered or signature forged

Why This Process Works

1. Hash Function Properties:
- Deterministic: Same input always produces same output
- One-way: Cannot reverse to get original input
- Avalanche effect: Small change produces completely different hash
- Collision resistant: Different inputs produce different hashes

2. Public Key Cryptography:
- Asymmetric: Different keys for encryption/decryption
- Mathematically related: Public and private keys are paired
- One-way trapdoor: Easy to encrypt with public key, hard to decrypt without private key

3. Security Guarantees:
- Authentication: Only someone with private key could create signature
- Integrity: Any change to document changes hash, invalidating signature
- Non-repudiation: Sender cannot deny creating signature (they have private key)

Hash Functions

What is a Hash Function?

A hash function is a mathematical algorithm that takes input of any size and produces a fixed-length output (hash value or digest).

Properties of Good Hash Functions

1. Deterministic:

Same input → Always same output
"Hello" → Always produces same hash

2. One-Way (Pre-image Resistance):

Hash → Cannot determine original input
Cannot reverse: hash → "Hello"

3. Avalanche Effect:

Small change → Completely different hash
"Hello" → Hash: ABC123...
"Hello!" → Hash: XYZ789... (completely different)

4. Collision Resistance:

Different inputs → Different hashes
"Hello" → Hash: ABC123...
"World" → Hash: DEF456... (different)

5. Fast Computation:

Hash computation is fast
Even for large documents

Common Hash Functions

SHA-256 (Secure Hash Algorithm 256-bit):
- Output: 256 bits (32 bytes)
- Widely used
- Considered secure
- Used in Bitcoin, TLS, etc.

SHA-512:
- Output: 512 bits (64 bytes)
- More secure than SHA-256
- Slower computation
- Used in high-security applications

SHA-3:
- Latest SHA standard
- Different design from SHA-2
- Multiple output sizes (224, 256, 384, 512 bits)

MD5 (Deprecated):
- Output: 128 bits
- Not secure - vulnerable to collisions
- Do not use for security applications

SHA-1 (Deprecated):
- Output: 160 bits
- Not secure - vulnerable to collisions
- Do not use for new applications

import hashlib

hash_hex = hash_object.hexdigest()
print(f"SHA-256: {hash_hex}")
# Output: a591a6d40bf420404a011733cfb7b190d62c65bf0bcda32b57b277d9ad9f146e# Small changemessage2 = "Hello World!"hash_object2 = hashlib.sha256(message2.encode())
hash_hex2 = hash_object2.hexdigest()
print(f"SHA-256: {hash_hex2}")
# Output: 7f83b1657ff1fc53b92dc18148a1d65dfc2d4b1fa3d677284addd200126d9069# Completely different!

Why Hash Before Signing?

1. Efficiency:
- Signing large documents is slow
- Hashing is fast
- Sign hash instead of entire document

2. Security:
- Hash is fixed length (e.g., 256 bits)
- Easier to sign fixed-length data
- Same security guarantees

3. Flexibility:
- Can sign documents of any size
- Hash always same size
- Algorithm works for any document

Public Key Cryptography

What is Public Key Cryptography?

Also known as asymmetric cryptography, it uses a pair of mathematically related keys:
- Private Key: Kept secret, used to sign
- Public Key: Shared publicly, used to verify

Key Properties

1. Key Pair Generation:
- Keys are generated together
- Mathematically related
- Cannot derive private from public

2. One-Way Function:
- Easy to compute in one direction
- Hard to reverse without private key
- Based on mathematical problems (factoring, discrete log)

3. Security:
- Private key must be kept secret
- Public key can be shared
- Compromise of private key = all signatures invalid

How It Works for Signing

Signing (Private Key):

Hash → Private Key → Signature
  • Only private key holder can create signature
  • Anyone can verify with public key

Verification (Public Key):

Signature → Public Key → Hash
  • Anyone can verify
  • Only private key could create signature

Mathematical Foundation

RSA Example:
- Based on factoring large numbers
- Private key: (d, n)
- Public key: (e, n)
- Signature: hash^d mod n
- Verification: signature^e mod n = hash

ECDSA Example:
- Based on elliptic curve discrete logarithm
- More efficient than RSA
- Smaller key sizes for same security
- Used in Bitcoin, Ethereum

Digital Signature Algorithms

RSA (Rivest-Shamir-Adleman)

Overview:
- One of first public-key cryptosystems
- Widely used and well-understood
- Based on factoring large numbers

Key Sizes:
- 2048 bits: Minimum recommended
- 3072 bits: Recommended for long-term security
- 4096 bits: High security

Pros:
- Widely supported
- Well-studied
- Flexible (can encrypt and sign)

Cons:
- Larger key sizes needed
- Slower than ECDSA
- Older algorithm

DSA (Digital Signature Algorithm)

Overview:
- Federal Information Processing Standard
- Based on discrete logarithm problem
- Used in government applications

Key Sizes:
- 1024 bits: Deprecated
- 2048 bits: Minimum
- 3072 bits: Recommended

Pros:
- Government standard
- Secure when properly configured

Cons:
- Less flexible than RSA
- Not as widely used
- Requires random number generation

ECDSA (Elliptic Curve Digital Signature Algorithm)

Overview:
- Based on elliptic curve cryptography
- More efficient than RSA/DSA
- Smaller key sizes for same security

Key Sizes:
- 256 bits: Equivalent to 3072-bit RSA
- 384 bits: Equivalent to 7680-bit RSA
- 521 bits: Very high security

Pros:
- Smaller keys
- Faster computation
- Lower resource usage
- Used in Bitcoin, Ethereum

Cons:
- More complex implementation
- Requires careful random number generation
- Newer (less historical use)

EdDSA (Edwards-Curve Digital Signature Algorithm)

Overview:
- Modern elliptic curve algorithm
- Based on twisted Edwards curves
- Deterministic (no random number needed)

Key Sizes:
- 256 bits: Ed25519
- 448 bits: Ed448

Pros:
- Deterministic (more secure)
- Fast
- Simple implementation
- Used in modern applications

Cons:
- Newer algorithm
- Less historical use

Digital Certificates and PKI

What is a Digital Certificate?

A digital certificate is an electronic document that binds a public key to an identity, verified by a Certificate Authority (CA).

Certificate Contents

Standard Fields:
- Subject: Entity the certificate belongs to
- Public Key: The public key being certified
- Issuer: Certificate Authority that issued it
- Validity Period: Start and end dates
- Serial Number: Unique identifier
- Signature: CA’s digital signature

Example:

Certificate:
  Subject: CN=example.com
  Public Key: RSA 2048-bit
  Issuer: CN=VeriSign
  Valid: 2024-01-01 to 2025-01-01
  Serial: 1234567890
  Signature: (signed by VeriSign)

Public Key Infrastructure (PKI)

Components:

  1. Certificate Authority (CA):
    • Trusted third party
    • Issues certificates
    • Verifies identities
    • Signs certificates
  1. Registration Authority (RA):
    • Verifies certificate requests
    • Validates identities
    • Submits to CA
  1. Certificate Repository:
    • Stores certificates
    • Makes them publicly available
    • LDAP, HTTP, etc.
  1. Certificate Revocation:
    • CRL (Certificate Revocation List)
    • OCSP (Online Certificate Status Protocol)
    • Revokes compromised certificates

Certificate Chain

How It Works:

Root CA Certificate
    ↓ (signed by)
Intermediate CA Certificate
    ↓ (signed by)
End Entity Certificate (Your certificate)

Verification:
1. Verify end entity certificate with intermediate CA
2. Verify intermediate CA with root CA
3. Root CA is trusted (in browser/OS trust store)

Why Certificates Are Needed

Without Certificate:
- Public key could be from anyone
- Man-in-the-middle attack possible
- No way to verify identity

With Certificate:
- CA vouches for identity
- Public key is verified
- Trust chain established
- Prevents MITM attacks

Creating Digital Signatures

Step-by-Step Process

1. Generate Key Pair:

from cryptography.hazmat.primitives.asymmetric import rsa
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.asymmetric import padding
# Generate private keyprivate_key = rsa.generate_private_key(
    public_exponent=65537,
    key_size=2048)
# Get public keypublic_key = private_key.public_key()

2. Hash the Document:

from cryptography.hazmat.primitives import hashes
message = b"Hello World"hasher = hashes.Hash(hashes.SHA256())
hasher.update(message)
hash_value = hasher.finalize()

3. Sign the Hash:

from cryptography.hazmat.primitives.asymmetric import padding
signature = private_key.sign(
    message,
    padding.PSS(
        mgf=padding.MGF1(hashes.SHA256()),
        salt_length=padding.PSS.MAX_LENGTH
    ),
    hashes.SHA256()
)

Complete Example:

from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.backends import default_backend
# 1. Generate key pairprivate_key = rsa.generate_private_key(
    public_exponent=65537,
    key_size=2048,
    backend=default_backend()
)
public_key = private_key.public_key()
# 2. Document to signdocument = b"This is an important document"# 3. Sign the documentsignature = private_key.sign(
    document,
    padding.PSS(
        mgf=padding.MGF1(hashes.SHA256()),
        salt_length=padding.PSS.MAX_LENGTH
    ),
    hashes.SHA256()
)
print(f"Signature: {signature.hex()}")

Verifying Digital Signatures

Step-by-Step Process

1. Receive Document and Signature:

received_document = b"Hello World"received_signature = b"...signature bytes..."

2. Hash the Received Document:

from cryptography.hazmat.primitives import hashes
hasher = hashes.Hash(hashes.SHA256())
hasher.update(received_document)
computed_hash = hasher.finalize()

3. Decrypt the Signature:

from cryptography.hazmat.primitives.asymmetric import padding
try:
    public_key.verify(
        received_signature,
        received_document,
        padding.PSS(
            mgf=padding.MGF1(hashes.SHA256()),
            salt_length=padding.PSS.MAX_LENGTH
        ),
        hashes.SHA256()
    )
    print("Signature is VALID")
except InvalidSignature:
    print("Signature is INVALID")

Security Considerations

1. Private Key Protection

Critical:
- Private key must be kept secret
- If compromised, all signatures are invalid
- Attacker can forge signatures

Protection Methods:
- Hardware Security Modules (HSM): Physical security
- Key Management Systems: Centralized management
- Encryption at Rest: Encrypt private keys
- Access Controls: Limit who can use private key
- Key Rotation: Regularly rotate keys

Best Practices:
- Never share private key
- Use strong passwords for key files
- Store in secure location
- Use hardware tokens when possible
- Implement key rotation policies

2. Hash Function Security

Important:
- Use secure hash functions
- SHA-256 minimum
- Avoid MD5, SHA-1

Recommendations:
- SHA-256: Good for most applications
- SHA-512: Higher security
- SHA-3: Latest standard
- Avoid: MD5, SHA-1 (deprecated)

3. Algorithm Selection

Considerations:
- Security requirements
- Performance needs
- Compatibility
- Key size

Recommendations:
- RSA-2048: General purpose
- RSA-3072: Long-term security
- ECDSA-256: Modern applications
- EdDSA-25519: Modern, fast

4. Certificate Management

Important:
- Verify certificate validity
- Check certificate revocation
- Validate certificate chain
- Monitor expiration dates

Best Practices:
- Use trusted CAs
- Implement certificate pinning
- Check revocation status (OCSP/CRL)
- Monitor certificate expiration
- Implement certificate rotation

5. Timestamping

Purpose:
- Prove when signature was created
- Legal validity
- Long-term verification

How It Works:

Document → Hash → Private Key → Signature
                              ↓
                    Timestamp Authority
                              ↓
                    Timestamped Signature

Benefits:
- Proof of signature time
- Validates signature after key expiration
- Legal compliance

6. Non-Repudiation

Ensuring Non-Repudiation:
- Proper key management
- Audit logs
- Timestamping
- Legal frameworks

Challenges:
- Key compromise
- Shared keys
- Legal requirements vary by jurisdiction

Summary

Key Takeaways

  1. Digital signatures provide:
    • Authentication (who sent it)
    • Integrity (not modified)
    • Non-repudiation (cannot deny)
  1. Process:
    • Hash document → Sign hash with private key → Attach signature
    • Verify: Hash document → Decrypt signature → Compare hashes
  1. Security depends on:
    • Private key protection
    • Secure hash functions
    • Strong algorithms
    • Proper implementation
  1. Applications:
    • Software distribution
    • Email security
    • Financial transactions
    • Legal documents
    • Code signing

Best Practices

✅ Do:
- Use strong hash functions (SHA-256 minimum)
- Protect private keys
- Use appropriate key sizes
- Implement proper key management
- Verify certificates
- Use timestamping for long-term validity

❌ Don’t:
- Use weak hash functions (MD5, SHA-1)
- Share private keys
- Use small key sizes
- Skip certificate validation
- Ignore key expiration

Remember

  • Digital signature ≠ Encryption
  • Hash is a step, not the signature
  • Private key signs, public key verifies
  • Certificates bind public keys to identities
  • Security depends on private key protection
